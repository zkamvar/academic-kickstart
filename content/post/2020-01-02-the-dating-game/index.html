Dates are a never-ending source of fresh hell. There are so many different ways parsing dates can go wrong. It‚Äôs not quite as bad as the horror that is parsing HTML with regex, but it‚Äôs close. This is only just the beginning. There are so many blog posts already dedicated to working with dates in R, and I‚Äôm not going to re-hash the whole rigormarole of explaining the difference between Date and POSIXt classes. I‚Äôm just going to point you to some excellent walkthroughs such as this gist by Bonnie Dixon (hosted by Noam Ross).

What I want to focus on is: what tools are available to parse date strings to the Date class in R and how well do they work to weird situations? Some of the common things I‚Äôve seen in my work for example:

  • an ambiguous mix of dd/mm/yy and mm/dd/yy format (e.g.¬†02/04/20 or 04/02/20)
  • Dates are in French, but you‚Äôre working in an English locale (e.g.¬†04 F√©vrier 2020)
  • Dates are imported from Excel (e.g.¬†43865, which are represented as the either the number of days since 1899-12-30 if you are importing from Windows OR the number of days since 1904-01-01 on MacOS)1

Setting up the {readr} function for use with {purrr}

the_locales <- c("en", "en", "en", "en", "fr", "en")
readr_parse_date <- function(date, format, locale) {
  readr::parse_date(date, format = format, locale = readr::locale(locale))
}
purrrlist <- list(the_dates, c(the_formats, NA), the_locales)
the_origin <- as.Date("1970-01-01")
res <- tibble(
  original      = the_dates,
  base          = as.Date(the_dates,                    format = the_formats),
  lubridate     = lubridate::parse_date_time(the_dates, orders = lub_formats),

There‚Äôs a lot going on here, so I‚Äôll summarize a few things:

  1. None of these format perfectly, locales and numeric dates are hard to handle.
  2. {datetimeutils} could not convert a single date presented and {rccdate} could only parse the ISO 8601 formatted date (but it‚Äôs not meant to be a general date parsing package).
  3. By far, the most successful package was {readr}, but it takes a bit of work to set up the function to return a date vector with multiple locales and formats.
  4. The most successful ‚Äúmagic‚Äù package (one without knowledge of formats) was {parsedate}.
  5. {anytime} is super conservative, but will assume that a string of numbers represents a date (e.g.¬†20200204).
  6. {lubridate} does fairly well, but it makes some strange mistakes with the month spelled out.
  7. {base} only does well here because I gave it the EXACT specifications for each date. If I didn‚Äôt, it would only be able to parse the ISO 8601 date.

No right answer

I think the main thing to take away from this exercise is that there is no right answer when it comes to parsing dates. I‚Äôve helped out with a project that aims at trying to provide yet another magic solution, but even it has drawbacks. You ultimately get the best results when you know the formats and locales you are dealing with, so the best option is just to evangelize about the ISO 8601 and hope. All the available packages have their own idiosyncrasies and it‚Äôs an absolute minefield when it comes to dates (note that this doesn‚Äôt even get into things like the year 2038 problem, or leap seconds).
